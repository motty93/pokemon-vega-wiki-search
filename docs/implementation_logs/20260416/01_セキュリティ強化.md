# セキュリティ強化（Cloud Run公開対応）

## 背景

Cloud Run で `--allow-unauthenticated` 公開している構成に対して、以下のセキュリティ対策が不足していた：

- HTTP サーバにタイムアウト設定がなく Slowloris 等に弱い
- レート制限なしで DoS 耐性が低い（`max-instances: 3` で容易に枯渇）
- CSP / HSTS 等のセキュリティヘッダ未設定
- SQLite にアナリティクス書き込みが走るが、Cloud Run のコンテナ FS は揮発＆非共有でほぼ機能不全
- Dockerコンテナが root で稼働
- Cloud Run 用の専用SAなし（デフォルトCompute SAの広い権限で稼働）
- Turso 認証情報が環境変数直挿し（Secret Manager 未使用）

## 変更内容

### アプリケーションコード

#### `cmd/server/main.go`
- `http.Server{}` を明示し `ReadHeaderTimeout=5s` / `ReadTimeout=10s` / `WriteTimeout=30s` / `IdleTimeout=120s` / `MaxHeaderBytes=1MB` を設定
- `httprate.LimitByRealIP` でレート制限を追加。全体 300/min、`/search` は 60/min（Cloud Run の `X-Forwarded-For` から実IP取得）
- `SecurityHeadersMiddleware` / `BodyLimitMiddleware(1MB)` を chi ミドルウェアチェーンに追加
- マイグレーション後に `EnableReadOnly(db)` を呼び出し SQLite を read-only 化
- `OpenAnalytics()` で Turso 接続を分離、nil の場合はアナリティクス no-op

#### `internal/handler/middleware.go`
- `SecurityHeadersMiddleware` 追加: CSP / HSTS / X-Frame-Options: DENY / X-Content-Type-Options: nosniff / Referrer-Policy / Permissions-Policy
  - CSP は Tailwind Play CDN が必要とする `'unsafe-eval'` を script-src に含む
- `BodyLimitMiddleware(maxBytes)` 追加: `http.MaxBytesReader` でボディサイズ制限

#### `internal/handler/handler.go`
- `Handler` 構造体に `AnalyticsDB *sql.DB` を追加
- `New(db, analyticsDB)` とシグネチャ変更
- `LogSearch` / `LogPageView` の呼び出し先を `h.DB` → `h.AnalyticsDB` に切り替え

#### `internal/db/db.go`
- `Open()` は従来通り読み取り用（SQLite優先）
- `OpenAnalytics()` 追加: Turso専用、`TURSO_URL` 未設定なら `(nil, nil)` 返却でアナリティクス無効化
- `EnableReadOnly(db)` 追加: `PRAGMA query_only=ON` で書き込み禁止
- `MigrateAnalytics(db)` 追加: Turso には `002_analytics.sql` のみ適用

#### `internal/db/analytics.go`
- `LogSearch` / `LogPageView` の先頭で `if db == nil { return }` を追加（analytics無効時の安全化）

### インフラ

#### `Dockerfile`
- `FROM scratch` → `FROM gcr.io/distroless/static-debian12:nonroot` に変更
- 全 `COPY --chown=nonroot:nonroot` で所有者を nonroot に
- `USER nonroot:nonroot` を明示
- ビルドフラグに `-trimpath -ldflags="-s -w"` を追加（バイナリ情報削減）

#### `cloudbuild.yml`
- Cloud Run デプロイ時に `--service-account vega-pokedex-runtime@$PROJECT_ID.iam.gserviceaccount.com` を指定
- `--set-secrets TURSO_URL=turso-url:latest,TURSO_AUTH_TOKEN=turso-auth-token:latest` で Secret Manager から注入

#### `scripts/setup-gcp.sh`
- ランタイムSA `vega-pokedex-runtime` の作成ロジック追加
- SA への最小権限 `roles/logging.logWriter` 付与
- Secret Manager `turso-url` / `turso-auth-token` を空で作成（版は別スクリプト）
- シークレットへの `roles/secretmanager.secretAccessor` を**リソースレベル**で付与
- Cloud Build SA に `roles/iam.serviceAccountUser` を付与（ランタイムSAのimpersonate用）
- ユーザ権限を `roles/storage.admin`（過剰） → `roles/artifactregistry.writer` に縮小
- `secretmanager.googleapis.com` / `iam.googleapis.com` を API 有効化に追加

#### `scripts/upload-secrets.sh`（新規）
- `.env` から `TURSO_URL` / `TURSO_AUTH_TOKEN` を読み取り Secret Manager にアップロード
- 既存値と同じなら新バージョン追加しない冪等動作

## 依存追加

- `github.com/go-chi/httprate v0.15.0` — IP単位レート制限

## 動作検証

- `go build ./...` / `go vet ./...` 成功
- ローカル起動し以下を `curl` で確認：
  - `/` にて CSP / HSTS / X-Frame-Options / X-Content-Type-Options / Referrer-Policy / Permissions-Policy 付与
  - `/search` を65回連射して60回目から HTTP 429 返却を確認
  - `/` は 300/min、`/search` は 60/min のレート制限が正しく分離

## 関連情報

- レート制限は chi の `Group` でスコープ分離。内側の `LimitByRealIP(60)` が外側の `LimitByRealIP(300)` と独立にカウントされ、レスポンスヘッダも内側の値で上書きされる
- SQLite read-only化は `PRAGMA query_only` なのでコネクション単位の設定。`database/sql` の接続プールでも全コネクションに適用される（モダンSQLiteドライバは `db.Exec` 実行時に全コネクションにPRAGMAが反映される）

## 残タスク
